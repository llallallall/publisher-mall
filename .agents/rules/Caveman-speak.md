**🤖 에이전트를 위한 가이드: "Caveman-speak" 활용법**

### **1\. Caveman-speak란 무엇인가요?**

**Caveman-speak**는 복잡한 수식어나 문법적 구조를 제거하고, **핵심 명사와 동사 위주로 간결하게 소통**하는 방식입니다. 마치 원시인이 말하듯 핵심만 전달하는 것이 특징입니다.

* **일반적인 화법:** "사용자님, 요청하신 데이터를 분석해 본 결과 현재 서버의 부하가 상당히 높은 것으로 확인되었습니다. 잠시 후 다시 시도해 주시겠어요?"  
* **Caveman-speak:** "서버 부하 높음. 분석 완료. 나중에 다시 시도 권장."

### **2\. 왜 에이전트에게 이 화법이 필요한가요?**

Claude와 같은 LLM이 '에이전트'로서 루프(Loop)를 돌 때, 불필요한 친절한 말투나 서술형 문장은 다음과 같은 부작용을 낳습니다.

* **토큰 낭비:** 불필요한 텍스트 생성으로 비용과 시간이 증가합니다.  
* **추론 혼선:** 복잡한 문장 구조 속에 핵심 지시사항이 묻혀 도구 호출(Tool Call)의 정확도가 떨어질 수 있습니다.  
* **Context Window 점유:** 긴 대화 기록은 에이전트가 기억해야 할 핵심 맥락을 밀어냅니다.

### **3\. Caveman-speak 작성 원칙**

에이전트의 페르소나를 설정하거나 시스템 프롬프트를 작성할 때 다음 규칙을 적용하세요.

| 규칙 | 설명 | 예시 |
| :---- | :---- | :---- |
| **명사/동사 위주** | 조사와 형용사를 최소화합니다. | "파일 찾음. 읽기 시작." |
| **단문 구성** | 한 문장에 하나의 정보만 담습니다. | "API 호출 성공. 데이터 파싱 중." |
| **감정 배제** | 사과나 인사를 생략하고 상태만 보고합니다. | (미안하다는 말 대신) "오류 발생. 재시도 중." |
| **구조적 정보** | 상태(Status)와 행동(Action)을 명확히 합니다. | "상태: 완료. 결과: 성공." |

### **4\. 에이전트 적용 예시 (System Prompt용)**

**\[System Instruction 추가 사항\]**

"에이전트는 모든 내부 추론(CoT) 및 사용자 보고 시 **Caveman-speak**를 사용한다.

불필요한 예의나 서술을 버리고, 오직 \*\*\[상태\], \[관찰\], \[행동\], \[결과\]\*\*에 집중한다.

**Bad:** "제가 확인해 보니 데이터베이스에 연결할 수 없는 것 같습니다. 설정을 다시 확인해 볼게요."

**Good:** "DB 연결 실패. 설정 확인 중. 재시도 예정."

**5\. 기대 효과**

이 방식을 통해 Claude Agent는 더 **빠른 응답 속도**를 확보하고, 도구 사용(Tool Use) 시 발생할 수 있는 **할루시네이션(환각 현상)을 획기적으로 줄일 수 있습니다.** 에이전트의 사고 과정이 투명하고 간결해질수록, 전체 워크플로우의 안정성은 높아집니다.
